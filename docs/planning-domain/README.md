# Planning Domain

This folder tracks planning knowledge extracted from runbooks and active implementation work.

Purpose:
- capture roadmaps, milestones, spike plans, and task sequencing
- separate planning from business requirements and design rationale
- support multiple candidate workstreams at the same time

Conventions:
- one Markdown file per roadmap, delivery track, or execution plan
- file names use a two-digit prefix and an expressive name
- examples: `01-litert-lm-gemma4-spike-and-delivery-roadmap.md`, `02-ios-catalyst-validation-roadmap.md`

Content rules:
- allow multiple competing roadmaps to coexist
- treat plans as mutable execution artifacts
- cross-reference business and design documents when a plan depends on them
